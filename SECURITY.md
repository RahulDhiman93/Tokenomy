# Security Policy

## Supported Versions

Tokenomy is currently in active development. Security updates are provided for the most recent stable release. Older versions may not receive fixes.

| Version | Supported |
| ------- | --------- |
| Latest  | ✅ |
| Older versions | ❌ |

We recommend always running the **latest release** to receive the newest security fixes and improvements.

---

## Reporting a Vulnerability

If you discover a security vulnerability in Tokenomy, please report it responsibly.

**Do not open a public GitHub issue for security vulnerabilities.**

Instead, report it privately by contacting the maintainers:

**Email:** security@tokenomy.dev  
(or open a private security advisory through GitHub)

Please include the following information in your report:

- A clear description of the vulnerability  
- Steps to reproduce the issue  
- Potential impact or exploit scenario  
- Any suggested fixes (if available)

---

## What to Expect

After a vulnerability report is submitted:

1. We will acknowledge receipt within **48 hours**.
2. The issue will be investigated and validated.
3. If confirmed, we will work on a fix and prepare a patch.
4. A security release will be published once the issue is resolved.

When appropriate, the reporter may be credited in the release notes for responsible disclosure.

---

## Scope

This security policy applies to:

- The Tokenomy core toolkit
- Hook implementations (PreToolUse and PostToolUse)
- Token optimization and trimming logic
- Token logging and reporting components

Issues related to external tools, third-party integrations, or user implementations may fall outside the scope of this policy.

---

## Responsible Disclosure

We ask researchers to:

- Avoid publicly disclosing the vulnerability until a fix is available
- Give maintainers reasonable time to address the issue
- Share enough information to allow reproduction and validation

We appreciate responsible disclosure and collaboration to help keep Tokenomy secure for everyone.
